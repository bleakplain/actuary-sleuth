# Feature Specification: RAG 记忆增强与 Agent 框架

**Feature Branch**: `004-rag-memory-agent`
**Created**: 2026-04-07
**Status**: Draft
**Input**: 重点关注 RAG 的检索和会话，补充记忆能力，引入 LangGraph + Mem0 实现智能体编排

## Background

当前 Actuary Sleuth 系统已有成熟的 RAG 引擎（向量+BM25 混合检索、Rerank、归因检测）和会话管理（SQLite 存储），但存在以下核心短板：

1. **无跨会话记忆** — 关闭页面就"失忆"，用户每次需重新描述上下文
2. **无记忆提取** — 对话结束后没有从对话中提取值得长期保存的信息
3. **无记忆管理** — 没有 ADD/UPDATE/DELETE/NOOP 的冲突处理机制
4. **无用户画像** — 没有结构化的用户偏好/认知体系
5. **硬编码流水线** — 当前是线性 RAG 流程，无法支持条件分支、多步规划等复杂工作流

## User Scenarios & Testing

### User Story 1 - 跨会话审核上下文延续 (Priority: P1)

精算师多次审核同类产品（如不同公司的重疾险），系统记住之前的审核结论和关注点，在新会话中自动关联历史审核上下文。

**Why this priority**: 精算师经常需要对比审核多个同类产品，跨会话记忆能显著减少重复工作，提升审核效率。

**Independent Test**:
1. 在会话 A 中审核"平安福重疾险"，讨论了等待期和免责条款的问题
2. 关闭会话 A，创建新会话 B
3. 在会话 B 中提问"之前审核的那个重疾险，等待期是怎么规定的？"
4. 系统能检索到会话 A 中的相关审核记录并给出准确回答

**Acceptance Scenarios**:

1. **Given** 精算师在会话 A 中审核了某产品并产生了审核结论, **When** 在新会话 B 中提到该产品或相关审核话题, **Then** 系统自动检索并注入历史审核记忆，回答中体现对历史上下文的理解
2. **Given** 精算师跨多个会话审核了 3 款同类产品, **When** 在新会话中要求对比这几款产品, **Then** 系统能从历史记忆中提取各产品的审核要点进行对比
3. **Given** 精算师之前标记某条款"需要关注", **When** 后续会话中再次遇到类似条款, **Then** 系统提醒该条款曾标记为需关注

---

### User Story 2 - 对话摘要记忆 (Priority: P1)

系统在每次对话结束后，自动从对话内容中提取值得长期保存的关键信息（审核结论、关注点、用户偏好等），存入长期记忆库。

**Why this priority**: 记忆提取是整个记忆系统的入口，提取质量直接决定长期记忆的有效性。

**Independent Test**:
1. 进行一次精算审核对话，讨论了某产品的免责条款和等待期问题
2. 对话结束后，系统自动提取记忆点
3. 查看记忆列表，确认提取了"免责条款相关结论"和"等待期相关结论"
4. 确认未提取"你好"、"谢谢"等无意义内容

**Acceptance Scenarios**:

1. **Given** 一次精算审核对话包含审核结论、法规引用和闲聊, **When** 对话结束触发记忆提取, **Then** 只提取审核结论和法规相关的结构化信息，忽略闲聊
2. **Given** 对话中精算师纠正了之前的一个判断（如"之前说等待期90天，实际应该是180天"）, **When** 记忆提取时检测到冲突, **Then** 系统执行 UPDATE 操作而非 ADD，避免矛盾记忆并存
3. **Given** 对话中提到"这个产品等待期是180天", **When** 该信息已存在于记忆库中且内容一致, **Then** 系统执行 NOOP 操作，不重复写入

---

### User Story 3 - 用户偏好记忆 (Priority: P1)

系统记住精算师的检索偏好（如常用条款类型、关注的风险领域），在后续检索中自动优化。

**Why this priority**: 精算师的专业领域不同（有人关注寿险、有人关注财险），个性化检索能提升结果相关性。

**Independent Test**:
1. 精算师在多次对话中频繁检索"免责条款"和"等待期"相关内容
2. 系统自动学习到该用户偏好关注这两个领域
3. 在后续检索中，系统自动提升"免责条款"和"等待期"相关结果的排序权重

**Acceptance Scenarios**:

1. **Given** 精算师在近 5 次对话中有 3 次检索了免责条款, **When** 精算师进行新的检索, **Then** 检索结果中与免责条款相关的内容排序提升
2. **Given** 精算师明确表示"我主要关注重疾险产品", **When** 后续检索时, **Then** 系统自动将重疾险相关的元数据过滤条件加入检索
3. **Given** 精算师的偏好发生变化（从重疾险转向财险）, **When** 新偏好信号出现多次, **Then** 用户画像中的偏好更新为最新领域

---

### User Story 4 - 记忆管理界面 (Priority: P2)

精算师可以查看、编辑和管理自己的记忆（查看记忆列表、删除错误记忆、手动添加偏好）。

**Why this priority**: 记忆可能提取错误或过时，用户需要有能力纠正。

**Independent Test**:
1. 打开记忆管理页面
2. 查看所有记忆条目，按类型（审核记忆/偏好/摘要）和时间排序
3. 删除一条错误记忆
4. 手动添加一条偏好"重点关注老年群体保险产品"

**Acceptance Scenarios**:

1. **Given** 系统已存储了多条用户记忆, **When** 精算师打开记忆管理页面, **Then** 展示所有记忆条目，包含内容摘要、类型标签、创建时间和来源会话
2. **Given** 精算师发现一条记忆信息有误, **When** 点击删除该记忆, **Then** 记忆被软删除，后续检索不再返回该条目
3. **Given** 精算师手动添加偏好信息, **When** 下次对话开始, **Then** 该偏好被注入到系统 Prompt 中

---

### User Story 5 - 记忆过期清理 (Priority: P2)

系统自动清理过期的记忆条目，保持记忆库精简有效。

**Why this priority**: 记忆只增不减会导致存储膨胀和检索噪声，需要自动清理机制。

**Independent Test**:
1. 设置某条记忆的 TTL 为 7 天
2. 7 天后，该记忆被自动标记为过期
3. 后续检索不再返回该条目

**Acceptance Scenarios**:

1. **Given** 一条记忆设置了 TTL 为 30 天, **When** 30 天过去后, **Then** 该记忆被自动软删除
2. **Given** 一条标记为"永久"的记忆（如用户基本信息）, **When** 自动清理任务运行, **Then** 该记忆不被删除
3. **Given** 一条记忆即将过期（剩余 3 天）, **When** 该记忆被检索命中, **Then** 该记忆的 TTL 自动续期（被使用=仍有价值）

---

### User Story 6 - Agent 工作流编排 (Priority: P2)

使用 LangGraph 编排审核工作流，支持条件分支和多步骤任务（如：检索 → 审核 → 生成报告）。

**Why this priority**: 当前是线性 RAG 流程，LangGraph 提供图编排能力，为未来复杂审核流程打基础。

**Independent Test**:
1. 定义一个审核工作流图：意图识别 → 记忆检索 → 知识库检索 → 记忆更新
2. 发送一个审核请求
3. 系统按图执行，每个节点完成后进入下一个节点
4. 可观测性系统记录每个节点的执行情况

**Acceptance Scenarios**:

1. **Given** 定义了审核工作流图, **When** 精算师发送审核请求, **Then** 系统按图定义的节点和边顺序执行，支持条件路由
2. **Given** 工作流中某个节点（如 LLM 调用）失败, **When** 框架检测到错误, **Then** 支持重试或回退到安全状态
3. **Given** 审核工作流包含记忆检索和记忆更新节点, **When** 对话执行, **Then** 记忆在对话开始时被检索注入，对话结束后新记忆被提取存储

---

### Edge Cases

- 用户在新会话中提到的产品与之前审核的产品名称略有不同（如"平安福" vs "平安福2019"），系统能否正确关联？
- 记忆提取 LLM 调用失败时，如何处理？（降级：跳过提取，不影响主流程）
- Mem0 服务不可用时，系统如何降级？（降级：不注入记忆，主 RAG 流程正常运行）
- 同一用户并发多个会话时，记忆写入是否有序？
- LangGraph 的 async 模型与现有同步 RAGEngine 的桥接是否可靠？

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 在每次对话开始时，自动检索与当前查询相关的用户历史记忆，注入到系统 Prompt 中
- **FR-002**: 系统 MUST 在每次对话结束后，自动从对话中提取值得长期记忆的信息（审核结论、用户偏好、关键事实）
- **FR-003**: 系统 MUST 使用 Mem0 管理记忆的 ADD/UPDATE/DELETE/NOOP 操作，处理记忆冲突
- **FR-004**: 系统 MUST 使用 LangGraph 编排审核工作流，支持条件路由和状态管理
- **FR-005**: 系统 MUST 按用户隔离记忆，不同用户的记忆不能互相检索
- **FR-006**: 系统 MUST 支持记忆的 TTL 过期机制，自动清理过期记忆
- **FR-007**: 系统 MUST 提供记忆管理 API（查看、删除、手动添加）
- **FR-008**: 系统 MUST 在 Mem0 不可用时降级运行，不影响主 RAG 流程
- **FR-009**: 系统 MUST 使用现有的 LLM 客户端（Zhipu/Ollama）进行记忆提取和管理
- **FR-010**: 系统 MUST 使用现有的存储（SQLite + LanceDB），不引入新的数据库
- **FR-011**: 系统 MUST 将记忆操作纳入现有的可观测性追踪系统（Trace/Span）

### Key Entities

- **Memory**: 用户的一条长期记忆，包含内容、类型（审核/偏好/摘要）、用户 ID、时间戳、TTL、来源会话
- **UserProfile**: 用户画像，包含基本信息、检索偏好、关注领域、历史统计
- **AgentState**: LangGraph 工作流状态，包含当前查询、检索结果、记忆上下文、LLM 响应
- **WorkflowNode**: 工作流节点定义，包含节点名称、处理函数、输入输出类型、条件路由

## Success Criteria

- **SC-001**: 跨会话记忆检索准确率 ≥ 80%（人工评估：检索到的记忆与当前查询的相关性）
- **SC-002**: 记忆提取精确率 ≥ 85%（提取的记忆中有价值信息占比）
- **SC-003**: 记忆冲突处理准确率 ≥ 90%（UPDATE/DELETE 操作的正确性）
- **SC-004**: 引入 Agent 框架后，现有 RAG 功能不受影响（所有现有测试通过）
- **SC-005**: Mem0 不可用时，系统在 1 秒内降级到无记忆模式
- **SC-006**: 记忆注入后，平均对话轮次减少（对比无记忆模式）

## Assumptions

- 用户（精算师）通过 Web UI 使用，每个用户有唯一标识（user_id）
- 现有 SQLite 数据库足以存储记忆元数据，Mem0 使用 SQLite 作为后端
- 现有的 Embedding 模型（Jina v5 / Zhipu embedding-3）可用于 Mem0 的向量检索
- 记忆提取使用现有的 QA LLM（glm-4-flash），不需要额外的 LLM 模型
- LangGraph 适配层通过 `asyncio.to_thread()` 桥接现有同步代码
- 当前阶段为单 Agent 架构，未来可扩展为多 Agent
- 记忆量级：单个用户的记忆条目预计在 100-1000 条范围内

## Technical Constraints

- **存储**: 保持 SQLite + LanceDB，不引入 Redis/PostgreSQL/Milvus 等新存储
- **LLM**: 保持现有 Zhipu/Ollama，不引入新的 LLM 提供商
- **依赖**: 新增 `langgraph`, `langchain-core`, `mem0ai`，总依赖增长控制在 20 个包以内
- **架构**: 遵循现有 `scripts/lib/` 模块结构，不创建新的顶级服务包
- **兼容**: 现有所有 API 端点和测试必须保持向后兼容
