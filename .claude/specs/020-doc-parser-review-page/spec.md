# Feature Specification: 产品文档解析结果审核页面

**Feature Branch**: `020-doc-parser-review-page`
**Created**: 2026-04-22
**Status**: Draft
**Input**: 系统梳理和学习了解doc_parser的整体设计和实现细节，重点review当前前端审核页面的设计与实现细节；我希望你参考知识库人工审核的设计，展示保险产品文档解析后的结果，方便用户进行人工验证

## User Scenarios & Testing

### User Story 1 - 查看解析结果 (Priority: P1)

作为精算师/审核人员，我希望查看产品文档解析后的结构化结果，以便快速了解文档提取的关键内容。

**用户旅程**：
1. 用户选择一个已解析的产品文档
2. 系统展示解析结果，包含条款、费率表、责任等结构化数据
3. 用户浏览各部分内容，了解文档核心信息

**Why this priority**: 查看是审核的基础，必须首先实现

**Independent Test**: 上传一个产品文档，解析后能正确展示所有提取的结构化字段

**Acceptance Scenarios**:

1. **Given** 文档已解析完成, **When** 用户打开审核页面, **Then** 显示所有解析结果（条款、费率表、责任、除外责任、健康告知等）
2. **Given** 解析结果包含多条条款, **When** 用户浏览条款列表, **Then** 每条条款显示编号、标题、正文
3. **Given** 解析结果包含费率表, **When** 用户查看费率表, **Then** 展示原始文本和结构化数据（如适用）

---

### User Story 2 - 对照原文验证 (Priority: P1)

作为精算师/审核人员，我希望逐项对照原文验证解析结果的准确性，以便发现和确认解析错误。

**用户旅程**：
1. 用户在解析结果中选择某一项（如某个条款）
2. 系统在原文中定位并高亮对应内容
3. 用户对比原文与解析结果，判断准确性

**Why this priority**: 对照验证是人工审核的核心价值，确保解析质量

**Independent Test**: 选择任意条款，原文自动定位到对应位置并高亮

**Acceptance Scenarios**:

1. **Given** 用户查看某条款, **When** 点击该条款, **Then** 原文滚动到对应位置并高亮显示
2. **Given** 用户查看费率表, **When** 点击费率表项, **Then** 原文定位到对应表格位置
3. **Given** 原文较长, **When** 用户切换不同解析项, **Then** 原文视图自动滚动到对应位置

---

### User Story 3 - 审核状态标记 (Priority: P2)

作为审核人员，我希望标记审核状态（通过/不通过/待定），以便记录审核结论并流转到下一环节。

**用户旅程**：
1. 用户完成对照验证后，选择审核结论
2. 系统记录审核状态
3. 用户可添加备注说明问题

**Why this priority**: 审核闭环需要状态流转，但可独立于查看/验证开发

**Independent Test**: 完成验证后能提交审核结论

**Acceptance Scenarios**:

1. **Given** 用户完成验证, **When** 选择"通过", **Then** 记录审核状态为通过
2. **Given** 用户发现解析错误, **When** 选择"不通过"并填写原因, **Then** 记录审核状态和原因

---

### User Story 4 - 开发调试视图 (Priority: P2)

作为开发人员，我希望查看解析过程的详细信息，以便调试和优化解析算法。

**用户旅程**：
1. 开发人员切换到"调试模式"
2. 查看解析过程的中间结果、耗时、置信度等信息
3. 定位解析问题的具体原因

**Why this priority**: 辅助功能，优先级低于业务审核需求

**Independent Test**: 调试模式下能查看解析元数据和中间结果

**Acceptance Scenarios**:

1. **Given** 开发人员进入调试模式, **When** 查看解析结果, **Then** 显示各字段的解析置信度
2. **Given** 解析过程有中间结果, **When** 展开详情, **Then** 显示解析链路和各阶段输出

---

- 超长文档的性能问题？
- 不支持的文档格式？

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 展示产品文档解析后的结构化结果，包括条款、费率表、责任、除外责任、健康告知等
- **FR-002**: 系统 MUST 支持用户选择解析结果中的某一项，并在原文中定位高亮对应内容
- **FR-003**: 系统 MUST 复用知识库审核页面的分块验证交互模式（左侧原文+右侧解析结果）
- **FR-004**: 系统 MUST 与现有审核流程集成，作为审核预处理结果的展示入口
- **FR-005**: 系统 SHOULD 支持标记审核状态（通过/不通过/待定）
- **FR-006**: 系统 SHOULD 支持添加审核备注
- **FR-007**: [NEEDS CLARIFICATION] 解析错误修正：是否支持在线编辑修正？修正后如何处理？

### Key Entities

- **AuditDocument**: 解析后的产品文档，包含 clauses（条款）、premium_tables（费率表）、notices（须知）、health_disclosures（健康告知）、exclusions（除外责任）、rider_clauses（附加险条款）等字段
- **Clause**: 单条条款，包含编号、标题、正文、位置信息
- **PremiumTable**: 费率表，包含原始文本、结构化数据（行列）
- **审核状态**: 审核结论和备注

## Success Criteria

- **SC-001**: 用户能查看所有解析结果字段
- **SC-002**: 用户点击解析项后，原文能在 1 秒内定位到对应位置
- **SC-003**: 审核状态能正确保存和展示

## Assumptions

- 文档解析结果已存储在数据库或文件中，可直接读取
- 原文可通过文档 URL 或本地路径访问
- 审核流程已有状态管理机制，本页面集成到现有流程中
- 前端技术栈与现有 web 项目一致（React + TypeScript）
