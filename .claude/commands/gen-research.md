# Gen Research - 代码库深度研究报告生成器

深入阅读当前工作区代码实现，深刻理解其工作原理、功能以及所有细节，了解其复杂性，并走通完整系统流程，排查潜在 bug。完成后，将学习和发现写成一份详细的 research.md 报告。

## 核心要求

> **深入分析代码库，理解系统架构、数据流、关键模块，识别潜在问题，生成包含代码示例和具体位置的研究报告。**
>
> **支持两种模式：SDD 模式（基于 spec.md 定向研究）和兼容模式（原有深度分析）。**

## 命令格式

```bash
/gen-research [focus]
```

## 用法

```bash
/gen-research              # 自动检测模式，生成 research.md
/gen-research lib/audit    # 聚焦分析 lib/audit 模块
```

---

## 模式检测

从当前 git branch 名提取 feature-name（如 `001-kb-search`），检查 `.claude/specs/<feature-name>/spec.md` 是否存在。

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **SDD 模式** | `.claude/specs/<feature-name>/spec.md` 存在 | 基于 spec.md 定向研究：分析现有代码如何支持需求实现、技术选型调研、依赖分析 |
| **兼容模式** | 无 spec.md | 询问 feature-name → 基于 origin/master 创建 worktree → 保持原有行为：深度代码分析、问题识别 |

---

## SDD 模式执行步骤

### 第一步：读取 spec.md

1. 读取 `.claude/specs/<feature-name>/spec.md`
2. 提取所有 User Stories、Functional Requirements、Key Entities
3. 识别需要研究的技术问题

### 第二步：现有代码分析

对于 spec.md 中的每个需求：

1. **定位相关模块** — 找到现有代码中与需求相关的模块
2. **分析可复用组件** — 可复用的类、函数、数据结构
3. **评估改动范围** — 需要新增/修改/删除的模块

### 第三步：技术选型研究

1. **方案对比** — 至少 2-3 种技术方案，列表格对比
2. **依赖分析** — 新增依赖、版本兼容性
3. **风险评估** — 技术假设需要验证的点

### 第四步：数据流分析

1. **现有数据流** — 追踪数据如何在模块间传递
2. **新增/变更的数据流** — 需求引入的新数据流
3. **关键数据结构** — 需要新增或修改的数据模型

### 第五步：生成研究报告

输出到 `.claude/specs/<feature-name>/research.md`，结构如下：

```markdown
# [Feature Name] - 技术调研报告

生成时间: YYYY-MM-DD HH:MM:SS
源规格: .claude/specs/<feature-name>/spec.md

## 执行摘要

简要概述调研发现、技术选型建议、风险提示（3-5 句话）。

## 兼容模式执行步骤

### 第一步：项目结构分析

1. **识别目录结构**
   - 列出所有主要目录
   - 识别模块边界
   - 分析依赖关系

2. **识别关键文件**
   - 入口文件（main.py, audit.py 等）
   - 核心业务逻辑文件
   - 配置文件
   - 测试文件

### 第二步：深入代码分析

对于每个关键模块：

1. **读取并理解代码**
   - 逐文件阅读
   - 理解类和函数的作用
   - 识别设计模式

2. **分析数据流**
   - 追踪数据如何在模块间传递
   - 识别数据转换点
   - 记录关键数据结构

3. **理解控制流**
   - 识别主要流程
   - 理解错误处理
   - 分析异步/并发逻辑

### 第三步：潜在问题识别

关注以下方面：

1. **安全漏洞** — SQL 注入、命令注入、XSS/CSRF、敏感信息泄露
2. **代码质量问题** — 重复代码、复杂度过高、命名不清晰、缺少错误处理
3. **性能问题** — N+1 查询、不必要的计算、资源泄漏
4. **设计缺陷** — 紧耦合、违反单一职责、缺少抽象

## 分析指南

### 安全检查清单

- [ ] 用户输入是否验证？
- [ ] SQL 查询是否使用参数化？
- [ ] 命令执行是否有注入风险？
- [ ] 敏感数据是否加密？
- [ ] 认证授权是否正确？
- [ ] 错误信息是否泄露信息？

### 代码质量检查清单

- [ ] 命名是否清晰？
- [ ] 函数是否过长？
- [ ] 是否有重复代码？
- [ ] 错误处理是否完善？
- [ ] 类型注解是否完整？
- [ ] 注释是否有用？

### 性能检查清单

- [ ] 是否有不必要的循环？
- [ ] 数据库查询是否优化？
- [ ] 是否有内存泄漏？
- [ ] 缓存策略是否合理？
- [ ] 并发处理是否安全？

### 设计检查清单

- [ ] 模块职责是否单一？
- [ ] 依赖方向是否正确？
- [ ] 接口是否稳定？
- [ ] 扩展性如何？
- [ ] 是否过度设计？

## 注意事项

1. **深入阅读**: 不要只看文件名，要实际读取代码内容
2. **理解上下文**: 理解代码为什么这样写，考虑历史原因
3. **具体位置**: 记录问题时，精确到文件和行号
4. **代码示例**: 引用实际代码片段，支持结论
5. **客观分析**: 区分问题和风格偏好
6. **可操作建议**: 提出具体可执行的改进建议
7. **产物位置**: 所有输出写入 `.claude/specs/<feature-name>/research.md`，不写项目根目录
