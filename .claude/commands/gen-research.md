# Gen Research - 代码库深度研究报告生成器

深入阅读当前工作区代码实现，深刻理解其工作原理、功能以及所有细节，了解其复杂性，并走通完整系统流程，排查潜在 bug。完成后，将学习和发现写成一份详细的 research.md 报告。

## 核心要求

> **深入分析代码库，理解系统架构、数据流、关键模块，识别潜在问题，生成包含代码示例和具体位置的研究报告。**

## 命令格式

```bash
/gen-research [focus]
```

## 用法

```bash
/gen-research              # 分析当前工作目录，生成 research.md
/gen-research lib/audit    # 聚焦分析 lib/audit 模块，生成 research.md
```

---

## 执行步骤

### 第一步：项目结构分析

1. **识别目录结构**
   - 列出所有主要目录
   - 识别模块边界
   - 分析依赖关系

2. **识别关键文件**
   - 入口文件（main.py, audit.py 等）
   - 核心业务逻辑文件
   - 配置文件
   - 测试文件

### 第二步：深入代码分析

对于每个关键模块：

1. **读取并理解代码**
   - 逐文件阅读
   - 理解类和函数的作用
   - 识别设计模式

2. **分析数据流**
   - 追踪数据如何在模块间传递
   - 识别数据转换点
   - 记录关键数据结构

3. **理解控制流**
   - 识别主要流程
   - 理解错误处理
   - 分析异步/并发逻辑

### 第三步：潜在问题识别

关注以下方面：

1. **安全漏洞**
   - SQL 注入
   - 命令注入
   - XSS/CSRF
   - 敏感信息泄露

2. **代码质量问题**
   - 重复代码
   - 复杂度过高
   - 命名不清晰
   - 缺少错误处理

3. **性能问题**
   - N+1 查询
   - 不必要的计算
   - 资源泄漏

4. **设计缺陷**
   - 紧耦合
   - 违反单一职责
   - 缺少抽象

### 第四步：生成研究报告

## 分析指南

### 安全检查清单

- [ ] 用户输入是否验证？
- [ ] SQL 查询是否使用参数化？
- [ ] 命令执行是否有注入风险？
- [ ] 敏感数据是否加密？
- [ ] 认证授权是否正确？
- [ ] 错误信息是否泄露信息？

### 代码质量检查清单

- [ ] 命名是否清晰？
- [ ] 函数是否过长？
- [ ] 是否有重复代码？
- [ ] 错误处理是否完善？
- [ ] 类型注解是否完整？
- [ ] 注释是否有用？

### 性能检查清单

- [ ] 是否有不必要的循环？
- [ ] 数据库查询是否优化？
- [ ] 是否有内存泄漏？
- [ ] 缓存策略是否合理？
- [ ] 并发处理是否安全？

### 设计检查清单

- [ ] 模块职责是否单一？
- [ ] 依赖方向是否正确？
- [ ] 接口是否稳定？
- [ ] 扩展性如何？
- [ ] 是否过度设计？

## 注意事项

1. **深入阅读**: 不要只看文件名，要实际读取代码内容
2. **理解上下文**: 理解代码为什么这样写，考虑历史原因
3. **具体位置**: 记录问题时，精确到文件和行号
4. **代码示例**: 引用实际代码片段，支持结论
5. **客观分析**: 区分问题和风格偏好
6. **可操作建议**: 提出具体可执行的改进建议
